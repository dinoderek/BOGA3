# Exercise Tags (Brainstorm)

## Purpose

Explore a lightweight tagging model for logged exercises so users can capture setup/context details that matter for later review and analytics.

This is a brainstorming document, not source of truth.

## Problem statement

Exercise names alone are not always enough to explain performance.

Examples:

- `Lat pulldown` done with a wide grip vs close neutral grip.
- The same machine exercise done on machine `#2` vs machine `#5`.
- A lift performed with straps, sleeves, or another repeatable setup choice.

The app needs a fast way to capture those distinctions during logging without forcing users to create duplicate exercise definitions.

## Working product direction

Recommended v1 direction:

- A tag is attached to a specific logged exercise instance inside a session.
- Tag suggestions are scoped to the underlying exercise definition, not global across the whole app.
- Tags are user-private.
- A logged exercise can have zero or more tags.
- The same tag cannot be attached twice to the same logged exercise.
- Tag definitions can be soft-deleted so they disappear from future suggestions without erasing historical usage.

This keeps the logging interaction fast while preserving meaningful per-exercise analytics later.

## Why tags should be exercise-scoped

The user intent in the initial notes is not “global labels for everything.” It is “remember the repeated variants I use for this exercise.”

Examples:

- `wide grip` means something useful for `Lat pulldown`.
- `machine 3` may be useful for `Seated leg curl`.
- Those same labels may be meaningless or misleading on unrelated exercises.

Because of that, the reusable tag library should be scoped per exercise definition.

## User stories

1. While logging an exercise, the user can quickly add one or more tags without leaving the recorder.
2. When adding a tag, the user can pick from tags they have used before for that exercise or create a new one inline.
3. The user can remove a tag from the current logged exercise quickly.
4. The user can manage tags for an exercise, including hiding deleted tags and restoring them later.
5. Analytics can later break down a single exercise by tag to answer questions like “How does my lat pulldown progress vary by grip?”

## Proposed UX shape

### Recorder exercise card

Conceptual layout:

```text
[Exercise name] [Add tag] [More]
[Tag chip] [Tag chip] [Tag chip]
[Set 1]
[Set 2]
[Add set]
```

Notes:

- Tags should render as compact chips under the exercise header.
- Each chip should have a fast remove affordance.
- `Add tag` should be available in both active logging and completed-session edit mode.

### Add tag flow

Recommended interaction:

```text
[Search / create input]
[Suggested tags for this exercise]
[Create "new tag" row when no exact match]
[Manage tags]
```

Behavior:

- Typing filters existing tags for the current exercise.
- Exact existing match selects that tag instead of creating a duplicate.
- If no exact match exists, the user can create a new tag inline.
- The modal should be optimized for one-hand rapid logging, not full administration.

### Manage tags flow

Recommended interaction:

```text
[Search input]
[Show deleted toggle]
[Tag row] [Delete or undelete]
[Close]
```

Behavior:

- Delete means “hide from future suggestions.”
- Deleted tags remain attached to historical logs where they were already used.
- Undelete restores them to the picker.

## Recommended domain model

Separate reusable tag definitions from per-session tag assignments.

### Tag definition

A reusable tag option for one exercise definition.

Conceptual fields:

- `id`
- `exercise_definition_id`
- `name`
- `deleted_at`
- `created_at`
- `updated_at`

Rules:

- Unique by `(exercise_definition_id, normalized_name)` among active tags.
- Soft-delete only.
- Rename is allowed.

### Tag assignment

A link between a logged session exercise and a tag definition.

Conceptual fields:

- `id`
- `session_exercise_id`
- `exercise_tag_definition_id`
- `created_at`

Rules:

- Unique by `(session_exercise_id, exercise_tag_definition_id)`.
- Removing a tag from a logged exercise deletes only the assignment, not the definition.

## Important current schema dependency

This feature wants tag suggestions to be scoped to an exercise definition, but the current local `session_exercises` schema appears to persist only:

- session linkage
- display name
- machine name
- origin scope/source metadata

It does not currently appear to persist a durable `exercise_definition_id` link.

That means this feature should not proceed as a pure UI task. It first needs one of these directions:

1. Add `exercise_definition_id` to `session_exercises`.
2. Define a different durable tag-scope key for logged exercises.

Recommendation: use a real `exercise_definition_id`. It is the cleanest fit for scoped suggestions and future analytics.

## Analytics semantics

Recommended v1 analytics posture:

- Tags are an optional breakdown dimension within a single exercise, not a global analytics taxonomy.
- Analytics should group tag usage only inside the same exercise definition.
- The app should not initially combine same-named tags across different exercises.

Examples:

- Good: `Lat pulldown` by `wide grip` vs `close neutral`.
- Good: `Leg press` by `machine 1` vs `machine 2`.
- Avoid in v1: global report for every `machine 1` tag across unrelated exercises.

## Historical behavior

Recommended v1 behavior:

- Tag assignments are historical facts on the logged exercise and do not disappear if the tag definition is later deleted.
- Tag definition rename/delete behavior should be retroactive at the metadata layer, consistent with current project direction for metadata semantics.

That means:

- rename updates history/analytics labels everywhere that tag definition is referenced;
- delete hides the tag from future selection, but historical assignments still render;
- undelete restores it for future selection without changing historical assignments.

## Validation and guardrails

Recommended v1 rules:

- Trim whitespace before save.
- Prevent blank tags.
- Prevent duplicate active tag names within the same exercise definition.
- Prevent duplicate assignment of the same tag on the same logged exercise.
- Preserve case for display, but compare normalized values case-insensitively.
- Keep names short enough to stay chip-friendly.

Suggested initial length limit:

- `1..32` visible characters.

## Out of scope for v1

- Global tag taxonomy across all exercises.
- Public/shared tags inside groups or brotherhoods.
- Tag colors, icons, categories, or hierarchy.
- Set-level tags.
- Auto-suggesting tags from machine/location context.
- Backend sync/API work.
- Cross-exercise analytics that treat same-named tags as equivalent.

## Option comparison

### Option A: Pure free-text tags per logged exercise

How it works:

- User types arbitrary text onto each logged exercise.
- No reusable tag definitions.

Pros:

- Lowest implementation cost.
- Very flexible.

Cons:

- No reliable suggestions.
- High spelling drift.
- Weak analytics grouping.
- No meaningful manage/delete/undelete flow.

Verdict:

- Too weak for the intended analytics and reuse behavior.

### Option B: Exercise-scoped reusable tag definitions plus per-log assignments

How it works:

- Each exercise definition owns its own reusable tag list.
- A logged exercise references one or more of those tags.

Pros:

- Matches the intended UX.
- Good analytics quality for a reasonable implementation cost.
- Supports soft-delete/undelete cleanly.

Cons:

- Requires schema work.
- Requires durable exercise-definition linkage from logged exercises.

Verdict:

- Recommended.

### Option C: Global reusable tag taxonomy

How it works:

- Tags are shared across all exercises.

Pros:

- Simpler reuse if the product later wants universal labels.

Cons:

- Poor semantic fit for exercise-specific variants.
- Higher risk of noisy tag lists.
- Harder to keep analytics interpretable.

Verdict:

- Not recommended for v1.

## Open questions

1. Should the UI cap the number of tags on one logged exercise for readability, or simply allow many and rely on user restraint?
2. Should the recorder show full tag-management UI inline, or route advanced management to the exercise catalog screen later?
3. Do we want tag rename in v1, or only delete/undelete?
4. Should tags be allowed on ad hoc exercises that are not tied to a saved exercise definition?
5. If a logged exercise is later changed from one exercise definition to another, what happens to its existing tag assignments?

Recommended answer for question 5:

- Remove assignments that no longer belong to the new exercise definition and ask for confirmation if any would be dropped.

## Suggested next decision slice

Before turning this into a milestone/task, lock these three points:

1. `session_exercises` will persist a durable `exercise_definition_id`.
2. Tags are scoped per exercise definition, not global.
3. Tag delete means soft-delete of the reusable definition, not deletion of historical usage.

Once those are locked, the feature can be split into:

1. local schema/repository work,
2. recorder UI for attach/remove,
3. manage-tags UI,
4. analytics follow-on work.
